北京华顺信安前端实习一面

2025.03.20 16:30

1. JS数据类型

1. 基本数据类型（Primitive Types）

  • Undefined：表示变量未初始化或未声明时的默认值。
  • Null：表示空值（一个特殊的空对象指针）。
  • Boolean：表示逻辑值：true 或 false。
  • Number：表示整数或浮点数（包括 Infinity、NaN 等特殊值）。
  • String：表示文本数据，用单引号、双引号或反引号包裹。
  • Symbol（ES6 新增）：表示唯一且不可变的值，常用于对象属性的键。
  • BigInt（ES2020 新增）：表示任意精度的整数，以 n 结尾。

2. 引用数据类型（Reference Types）

  • Object
    键值对的集合，包括以下常见子类：

    • Array（数组）

    • Function（函数）

    • Date（日期）

    • RegExp（正则表达式）

    • Map（键值对集合）

      • 特点：

        • 键可以是任意数据类型（对象、函数、基本类型等）。
        • 保持插入顺序，支持迭代。
        • 提供专为键值对操作设计的方法（如 set、get、has、delete）。

      • 与普通对象的区别：

        特性 普通对象 Map
        键的类型 只能是字符串或 Symbol 任意类型（包括对象、函数）
        键的顺序 无序（ES6 后按插入顺序遍历） 严格按插入顺序维护
        性能 适合少量、简单键值对 频繁增删键值对时性能更优
        序列化 支持 JSON.stringify 需手动转换为数组或对象

    • Set（唯一值集合）

      • 特点：

        • 成员唯一，自动去重（基于值的严格相等性）。
        • 保持插入顺序，支持迭代。
        • 提供专为集合操作设计的方法（如 add、has、delete）。

      • 与数组的区别：

        特性 数组 Set
        成员唯一性 允许重复元素 自动去重
        查找性能 O(n)（遍历查找） O(1)（哈希表实现）
        操作方法 push、pop、splice 等 add、delete、has 等

3. 关键区别

特性 基本数据类型 引用数据类型
存储方式 值直接存储在栈内存中 值存储在堆内存中，栈中存储堆地址引用
比较方式 按值比较（值相等即相等） 按引用比较（引用同一对象才相等）
示例 1 === 1 → true {} === {} → false
可变性 不可变（创建后不能修改） 可变（可修改属性或元素）

2. Map与普通对象的区别

在 JavaScript 中，Map 是 ES6 新增的数据结构，用于存储键值对，而普通对象（Object）是传统的键值对集合。以下是两者的核心区别：

1. 键的类型

  • 普通对象：键只能是 字符串（String） 或 Symbol，其他类型的键会被自动转换为字符串。
  • Map：键可以是 任意类型（包括对象、函数、NaN、undefined 等），且不会隐式转换。

2. 键的顺序

  • 普通对象：
    • 在 ES6 之前，对象的属性顺序不固定（不同引擎实现可能不同）。
    • ES6 及之后，对象属性按 插入顺序 遍历，但以下情况例外：
      • 数值键（如 "1"）会按数值大小升序排列。
  • Map：
    • 严格按 插入顺序 维护键值对的顺序。

3. 性能差异

  • 增删键值对：
    • Map 在频繁增删的场景下性能更优，因为它专为动态数据设计。
    • 普通对象需要处理原型链和哈希冲突，性能略低。
  • 查找速度：
    • 两者在查找键时性能接近（哈希表实现），但 Map 的 has 方法更直观。

4. 大小获取

  • 普通对象：
    需要手动计算键的数量（如 Object.keys(obj).length）。
  • Map：
    直接通过 size 属性获取键值对数量。

5. 迭代与遍历

  • 普通对象：
    • 需要先获取键数组（Object.keys、Object.entries），再遍历。
    • 无法直接使用 for...of。
  • Map：
    • 原生支持迭代，可直接使用 for...of 或 forEach。

6. 默认键的冲突

  • 普通对象：
    • 继承自 Object.prototype，可能包含默认属性（如 toString、constructor）。
    • 若自定义键与原型属性同名，可能引发意外覆盖。
  • Map：
    • 纯粹存储数据，无默认键，避免冲突。

7. 序列化与反序列化

  • 普通对象：
    • 支持 JSON.stringify 和 JSON.parse。
  • Map：
    • 无法直接序列化为 JSON，需手动转换为数组或对象。

总结：如何选择？

场景 推荐使用
键需要是非字符串类型（如对象、函数） Map
需要严格维护插入顺序 Map
高频增删键值对 Map
需要快速获取键值对数量 Map
需要 JSON 序列化 普通对象
简单键值对存储（如配置项） 普通对象

3. 实现对象的私有属性

在 JavaScript 中，对象没有原生真正的私有属性，但可以通过以下方法模拟私有属性的效果，以下是不同方案的实现和对比：

1. 闭包 + IIFE（传统方式）

利用闭包和立即执行函数隐藏变量，仅通过公共方法暴露访问权限。

缺点：

  • 所有实例共享同一个私有变量（如 _privateAge），无法实现实例隔离。

2. WeakMap 存储私有属性（ES6+）

通过 WeakMap 将私有数据与实例绑定，实现真正的实例隔离。

优点：

  • 实例销毁后，WeakMap 自动释放内存。
  • 每个实例拥有独立的私有属性。

3. ES2022 私有字段（# 语法）

使用 # 前缀声明真正的私有字段，现代浏览器和 Node.js 支持。

优点：

  • 语法简洁，符合直觉。
  • 真正的运行时私有，无法通过反射（如 Object.getOwnPropertySymbols）访问。
    兼容性：需现代环境支持（Chrome 84+、Node.js 12+）。

4. TypeScript 的 private 修饰符

TypeScript 提供编译时的私有属性检查，但编译后的 JS 代码中属性仍为公有。

注意：

  • private 仅在编译阶段检查，实际运行时可被访问。
  • 若需运行时私有，需结合 # 语法。

5. Symbol 键（伪私有）

用唯一 Symbol 作为属性键，减少命名冲突风险。

优点：

  • 避免属性名冲突。
  • 语法更简洁。

缺点：

  • 非真正私有，通过 Object.getOwnPropertySymbols() 可获取 Symbol 键。
  • 需手动管理 Symbol 的引用。

6. 命名约定（非强制）

通过 _ 前缀约定私有属性，依赖开发者自觉性。

优点：

  • 简单直观，适合团队协作规范。

缺点：

  • 无强制约束，易被误操作。

7. Proxy 代理拦截（高级方案）

通过 Proxy 拦截属性访问，隐藏私有字段（如 _ 前缀属性）。

适用场景：

  • 需要动态控制属性访问权限的复杂对象。

缺点：

  • 性能开销略高，且无法完全阻止通过原始对象访问私有属性。

JavaScript 早期缺乏原生的私有属性支持，导致社区采用各种模拟方案（如 Symbol 键、命名约束）。随着 ES6 引入 Symbol、WeakMap 和 ES2022 的 # 语法，语言逐步提供了更完善的解决方案。原生私有字段的推出，正是为了解决这些模拟方案的缺陷，因此应优先使用现代语法。

4. 事件循环

1. 事件循环的核心概念

JavaScript 是单线程语言，事件循环是其处理异步操作的核心机制。它通过协调 调用栈（Call Stack）、任务队列（Task Queue） 和 微任务队列（Microtask Queue） 来确保非阻塞执行。

2. 事件循环的工作流程

  1. 执行同步代码：
    • 同步代码按顺序执行，填充调用栈。
    • 遇到异步操作（如 setTimeout、Promise），将其回调交给 Web API 处理。
  2. 处理异步任务：
    • 宏任务（Macrotask）：如 setTimeout、setInterval、I/O 操作等，完成后回调进入 任务队列。
    • 微任务（Microtask）：如 Promise.then、MutationObserver、queueMicrotask，完成后回调进入 微任务队列。
  3. 事件循环的优先级：
    • 每轮事件循环 先清空 微任务队列，再执行 任务队列中的一个宏任务。
    • 重复此过程，直到所有队列为空。

3. 代码执行顺序示例

解析：

  1. 同步代码按顺序执行（输出 1 和 2）。
  2. Promise.then 回调进入微任务队列，setTimeout 回调进入宏任务队列。
  3. 同步代码执行完毕后，事件循环先处理微任务队列（输出 3 和 5），再处理宏任务队列（输出 4）。

4. 宏任务 vs 微任务

特性 宏任务（Macrotask） 微任务（Microtask）
常见来源 setTimeout、setInterval、I/O Promise.then、MutationObserver
执行时机 每轮事件循环执行一个宏任务 每轮事件循环清空微任务队列
优先级 低 高（优先执行）

5. 事件循环的阶段（Node.js 环境）

Node.js 的事件循环分为多个阶段，按顺序执行：

  1. Timers：执行 setTimeout、setInterval 回调。
  2. Pending Callbacks：处理系统操作（如 TCP 错误）的回调。
  3. Idle/Prepare：内部使用。
  4. Poll：检索新的 I/O 事件，执行相关回调。
  5. Check：执行 setImmediate 回调。
  6. Close Callbacks：执行关闭事件的回调（如 socket.on('close')）。

  1. 问题：微任务嵌套导致阻塞

  2. 问题：宏任务延迟不可靠

7. 总结

  • 事件循环优先级：同步代码 → 微任务 → 宏任务 → 渲染（如 requestAnimationFrame）。
  • 关键规则：
    • 微任务队列优先级高于宏任务队列。
    • 每轮事件循环处理完所有微任务后，才执行一个宏任务。

实用技巧：

  • 避免在微任务中执行耗时操作，防止阻塞主线程。
  • 合理使用 Promise 和 async/await 管理异步流程。

5. Vue2到Vue3的更新

从 Vue2 到 Vue3 的核心更新与迁移指南

一、Vue3 的核心改进

Vue3 于 2020 年发布，在性能、开发体验和扩展性上进行了全面优化，主要改进包括：

特性 Vue2 Vue3
响应式系统 Object.defineProperty Proxy（支持动态属性、数组索引修改）
代码组织 Options API（data, methods） Composition API（setup 函数）
虚拟 DOM 全量 Diff 静态标记 + 动态追踪（性能提升）
Tree-shaking 不支持 支持，按需引入（体积减少 40%+）
TypeScript 支持 需额外配置 原生支持，类型推导更完善
新特性 无 Fragment、Teleport、Suspense 等

二、主要变化详解

1. Composition API：更灵活的代码组织
  • Vue2 的 Options API：将代码按选项（data、methods、computed）拆分，逻辑分散。
  • Vue3 的 Composition API：通过 setup 函数按逻辑聚合代码，支持复用逻辑（类似 React Hooks）。

优势：

  • 逻辑复用更简单（自定义 Hook）。
  • TypeScript 支持更好。
2. 响应式系统的升级
  • Vue2 的局限性：
    • 无法检测对象属性的动态添加/删除（需 Vue.set/Vue.delete）。
    • 数组索引修改和 length 变化无法触发更新。
  • Vue3 的 Proxy 实现：
    • Proxy 直接代理整个对象而非对象属性，这样只需做一层代理就可以监听同级结构下的所有属性变化，包括新增属性和删除属性。
    • 惰性代理：仅在访问对象属性时递归代理嵌套对象，减少初始化开销。
3. 性能优化
  • 虚拟 DOM 优化：
    • 编译时标记静态节点，Diff 时跳过静态内容。
    • 动态节点绑定信息缓存，减少运行时计算。
  • Tree-shaking：
    默认仅打包使用到的功能（如 v-model 未使用则不包含）。
  • SSR 性能提升：
    服务端渲染速度提高 2-3 倍。
4. 新特性
  • Fragment：支持多根节点组件。
  • Teleport：将组件渲染到 DOM 任意位置（如模态框）。
  • Suspense：异步组件加载状态管理。

三、升级收益总结

  • 性能提升：更快的渲染、更小的包体积。
  • 开发体验优化：Composition API 提升代码可维护性。
  • 长期支持：Vue2 于 2023 年底停止维护，Vue3 是未来方向。

6. 打包工具Webpack，Vite

1. 核心设计理念

  • Webpack：
    • 目标：模块化打包工具，解决复杂项目的资源依赖和代码分割问题。
    • 工作流程：构建时打包所有资源，生成静态文件（Bundle）。
    • 适用场景：大型项目、需要高度定制化配置的场景。
  • Vite：
    • 目标：基于现代浏览器原生 ESM（ES Modules）的前端工具链，提升开发体验。
    • 工作流程：开发时按需编译和加载，生产时使用 Rollup 打包。
    • 适用场景：快速启动、轻量级项目、Vue/React 现代框架项目。

2. 核心差异对比

特性 Webpack Vite
构建理念 基于 打包（Bundle） 基于 原生 ESM（浏览器直接加载模块）
开发模式启动速度 慢（需提前打包所有依赖） 极快（仅启动服务器，按需编译）
热更新（HMR） 较慢（需重新构建模块关系） 极快（仅更新修改的模块）
配置复杂度 高（需手动配置 Loader/Plugin） 低（开箱即用，预设最佳实践）
生态成熟度 极高（丰富的插件和社区支持） 快速成长（插件生态逐渐完善）
适用场景 复杂项目、历史遗留项目 现代浏览器项目、轻量级应用
对 TypeScript 支持 需配置 ts-loader 或 babel-loader 原生支持（零配置）

3. 工作原理

  • Webpack：

    1. 依赖分析：从入口文件递归分析所有依赖。
    2. 打包构建：将所有模块合并为 Bundle（如 main.js）。
    3. 开发服务器：启动 DevServer，监听文件变化并重新打包。

    问题：随着项目规模增大，启动和热更新速度显著下降。

  • Vite：

    1. 开发模式：
      • 浏览器直接加载原生 ESM 模块。
      • 按需编译：仅编译当前页面所需模块（如 .vue、.ts 文件）。
    2. 生产模式：
      • 使用 Rollup 打包（支持 Tree-shaking、代码分割）。

    优势：开发阶段无需打包，冷启动和热更新极快。

4. 优缺点总结

  • Webpack：
    • 优点：功能强大、生态完善、适合复杂场景。
    • 缺点：配置复杂、开发体验较慢。
  • Vite：
    • 优点：开发体验极佳、配置简单、对现代框架支持好。
    • 缺点：生产构建需依赖 Rollup、插件生态尚不成熟。

5. 如何选择？

  • 选择 Webpack：
    • 项目依赖复杂（如多入口、自定义 Loader）。
    • 需要深度优化生产包（如自定义代码分割策略）。
    • 依赖 Webpack 生态插件（如 webpack-bundle-analyzer）。
  • 选择 Vite：
    • 追求极速开发体验（如频繁修改代码的敏捷开发）。
    • 使用 Vue/React 等现代框架的轻量级项目。
    • 希望减少配置成本，快速启动项目。

7. Webpack核心模块配置

Webpack 是一个模块打包工具，通过配置不同的核心模块，可以灵活处理项目的资源依赖、代码转换和优化。以下是 Webpack 的核心配置模块及其作用：

1. 入口（Entry）

作用：指定应用程序的入口文件，Webpack 从该文件开始构建依赖图。

2. 输出（Output）

作用：定义打包后的文件输出位置和命名规则。
常用配置项：

  • filename：输出文件名（支持 [name]、[hash] 等占位符）。
  • path：输出目录的绝对路径。
  • publicPath：资源引用路径（如 CDN 地址）。

3. 加载器（Loaders）

作用：处理非 JavaScript 文件（如 CSS、图片、字体等），将其转换为有效模块。
常见 Loader 配置：

  • 处理 CSS
  • 处理图片和字体
  • 转译 JavaScript（Babel）

4. 插件（Plugins）

作用：扩展 Webpack 功能，如生成 HTML 模板、压缩代码、提取 CSS 等。
常用插件：

  • HtmlWebpackPlugin：自动生成 HTML 文件并注入打包后的资源。
  • MiniCssExtractPlugin：将 CSS 提取为独立文件。
  • CleanWebpackPlugin：每次构建前清理输出目录。

5. 模式（Mode）

作用：设置构建模式，默认值为 production，影响内置优化策略。
可选值：

  • development：启用开发工具（如 source map），不压缩代码。
  • production：启用代码压缩、Tree-shaking 等优化。

6. 解析（Resolve）

作用：配置模块解析规则，简化路径引用。
常用配置项：

  • extensions：自动解析的文件扩展名（如 .js、.vue）。
  • alias：路径别名（简化导入路径）。

7. 开发服务器（DevServer）

作用：提供本地开发服务器，支持热更新（HMR）。

8. 优化（Optimization）

作用：配置生产环境下的代码优化策略。
常用配置项：

  • splitChunks：代码分割（拆分公共代码和第三方库）。
  • minimizer：自定义压缩工具（如 Terser 压缩 JS）。

  2. 如何优化构建速度？

高频面试题

  1. Loader 和 Plugin 的区别是什么？
    • Loader：处理单个文件的转换（如转译 JS、解析 CSS）
    • Plugin：在构建流程中执行更广泛的任务（如资源管理、环境注入）
  2. 如何优化 Webpack 构建速度？
    • 使用 cache 配置（Webpack5+ 文件系统缓存）
    • 并行处理：thread-loader、HappyPack
    • 缩小搜索范围：resolve.modules、resolve.alias
    • 预编译依赖：DLLPlugin（适合长期不变的第三方库）
  3. 代码分割（Code Splitting）有哪些方式？
    • 入口点：entry 配置多个入口
    • 动态导入：import() 语法
    • SplitChunksPlugin：抽离公共模块

总结：Webpack 的核心配置围绕入口、输出、加载器、插件和优化展开，合理配置这些模块可以显著提升开发效率和项目性能。根据项目需求灵活调整配置，结合社区插件（如 webpack-bundle-analyzer 分析包体积），能更好地掌控构建流程。

8. 新技术：新框架、新工具

一、新框架：突破传统开发模式

  1. Qwik
    • 核心特点：
      • 延迟加载（Lazy Loading）：实现“零 JavaScript 初始加载”，按需加载代码块。
      • 极致性能：适合超大型应用，首屏加载时间减少 70% 以上。
    • 适用场景：电商、内容型网站（如新闻门户）。
  2. Astro
    • 核心特点：
      • 岛屿架构（Islands Architecture）：静态页面中按需激活交互组件。
      • 多框架支持：可混合使用 React、Vue、Svelte 组件。
    • 适用场景：内容优先的网站（如文档站、营销页）。
    • 优势：首屏无 JavaScript，SEO 友好，交互组件按需加载。

二、新工具：颠覆开发体验

  1. Turbopack（Webpack 作者新作）
    • 优势：
      • 增量构建速度比 Webpack 快 700%，比 Vite 快 10 倍。
      • 支持 React Server Components 和 Next.js 深度集成。
    • 适用场景：超大型 Monorepo 项目（如企业级应用）。
  2. Bun（全栈运行时）
    • 功能整合：
      • 替代 Node.js：内置支持 TypeScript、JSX、SQLite。
      • 极速启动：比 Node.js 快 4 倍，包管理比 npm 快 30 倍。

三、新语言与标准

  1. TypeScript 5.0+ 新特性
    • 装饰器标准化：ECMAScript 正式装饰器提案落地。
    • satisfies 运算符：更安全的类型断言。

四、新兴技术趋势

  1. AI 工程化
    • 工具链：
      • Vercel AI SDK：快速集成 GPT、LangChain 到前端。
      • TensorFlow.js 4.0：浏览器内训练小型模型，支持 WebGPU 加速。
    • 应用场景：
      • 智能表单验证（语义分析）。
      • 实时语音/图像识别（如医疗影像辅助诊断）。
  2. 边缘计算（Edge Computing）
    • 技术栈：
      • Next.js Edge Runtime：在 CDN 边缘节点运行 Server Components。
      • Deno Deploy：全球分布式 JS 运行时，延迟低于 50ms。
    • 优势：用户就近处理数据，合规性（如 GDPR）更易实现。

总结：前端技术正朝着更高性能、更低功耗、更智能化的方向演进。开发者需关注工具链革新（如 Bun、Turbopack）、新架构范式（如 Islands Architecture）、以及 AI 的深度集成，才能在快速变化的技术浪潮中保持竞争力。
